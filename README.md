# TodoList Server

Go backend for `sym191/todolist-client`. The client remains local-first: every edit is committed to Drift first, written to an outbox, pushed idempotently, and then reconciled with the server's canonical record.

## Included

- Email/password registration and login
- Short-lived JWT access tokens
- Rotating, server-stored refresh tokens and per-device sessions
- PostgreSQL persistence and automatic ordered migrations
- Projects, tasks, custom board columns, soft deletion, and full task metadata
- Idempotent mutation push using client mutation IDs
- Incremental pull using a server-assigned monotonic revision cursor
- Full snapshot endpoint for a new installation
- Graceful shutdown, structured logs, request limits, CORS, and health endpoints
- OpenAPI contract, Docker image, Compose development stack, and GitHub Actions

## Run locally

```bash
docker compose up --build
```

The API listens on `http://localhost:8080`. Migrations run automatically before the server begins accepting traffic.

Register a development account:

```bash
curl -X POST http://localhost:8080/api/v1/auth/register \
  -H 'Content-Type: application/json' \
  -d '{"email":"dev@example.com","password":"development-password","deviceName":"Desktop"}'
```

Copy `.env.example` to `.env` when running the binary outside Compose. Production must use a randomly generated `JWT_SECRET` and a TLS-terminating reverse proxy.

## Sync contract

1. The client commits a local edit and an outbox row in the same Drift transaction.
2. `POST /api/v1/sync/push` sends up to `SYNC_BATCH_SIZE` mutations. A mutation ID is accepted only once per user.
3. The server applies mutations in request order. For concurrent edits to the same entity, the last mutation accepted by the server is canonical.
4. The push response contains that canonical entity and its server revision.
5. `GET /api/v1/sync/pull?cursor=N` returns ordered changes after `N`. The client applies them transactionally and persists the returned cursor.
6. A new empty client may use `GET /api/v1/sync/snapshot`. An existing offline client pushes its outbox before its first pull so local data is preserved.

See [api/openapi.yaml](api/openapi.yaml) for the wire contract.

## Development

```bash
go test ./...
go vet ./...
```

With PostgreSQL running from Compose:

```bash
make test-integration
```

The integration test applies migrations, registers an isolated user, pushes project/task mutations, verifies idempotency, pulls changes, and verifies a snapshot.

## Conflict and deletion rules

- IDs are generated by clients and remain stable across devices.
- Server versions and revisions are authoritative; client wall clocks do not decide conflicts.
- Deletes are tombstones, not physical removal, so offline clients can learn about deletions.
- Mutation IDs provide retry safety after timeouts or process crashes.
- A deleted project hides its children; clients should remove or hide local tasks and columns when applying a project tombstone.
